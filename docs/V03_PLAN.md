# Dyncamelo v0.3 Node Catalog Additions — "Plugin Parity" wave

Docs-ready addition to `docs/NODE_LIBRARY.md`. All rows follow the existing conventions: `Category.Verb` naming, `List<ModelItem>` as the lingua franca, zero-touch `public static` methods unless marked *(NodeModel)*, `document` ports defaulting to the active document, pass-through outputs for chaining, lengths in document units (convert with `Units.Convert`), errors → node Error state.

**Feasibility legend** — *Feasible (verified)*: every API member present in the 2024 MetadataLoadContext dumps AND exercised in the compile-proof (`scratchpad/lab/compiletest/Recipes.cs`, compiles green net48 against real DLLs). *Feasible (runtime-check)*: signature-verified, one behavioral assumption needs an in-Navisworks smoke test. *Partial*: works with documented limitations. *Spike*: route plausible but unverified — timebox before committing.

**Priority** — P1: unlocks multiple flagship plugin workflows or a wishlist item with a verified route. P2: strong single-workflow value or carries a runtime-check. P3: convenience / long tail.

New categories introduced: `Navisworks.Transform`, `Navisworks.Comments`, `Navisworks.Exchange`, `Navisworks.Grids` (existing tree already grew `Audit`/`Takeoff` in v0.2, so new categories are precedented).

## Risks

- **COM main-thread discipline** (Phases 2, 4, 5): all COM + .NET API calls must run on the UI thread — already a v0.2 invariant, but SetCustom over thousands of items needs progress/cancel and per-batch transactions to keep the UI alive.
- **Perf ceilings**: SetCustom is per-item COM round-trips (batch + progress); TryCalculateMinimumClearance on huge selections (document + cap); BCF snapshots at scale (stream, dispose Bitmaps promptly — System.Drawing keeps Dyncamelo.Navisworks net48-only, already true).
- **Copy semantics footguns** (Phase 3/5): centralized in I2; every write node must re-fetch stored items — code-review checklist item.
- **Schedule buffer**: Viewpoint.SetSectionBox (spike) and the P3 batch are the designated cuts; BCF.ImportIssues can slip to v0.3.1 without breaking the export leg.

## Explicit SKIP list (user-facing wording for release notes)

- **Autodesk Quantification**: Autodesk ships no API for the takeoff database — Dyncamelo can't drive it. Use `Takeoff.*` + `Excel.WriteToFile` to produce the same workbook-shaped deliverables.
- **IFC/DWG export**: Navisworks 2024 has no export API for these formats. (A rough glTF/OBJ exporter via COM mesh primitives is a v0.4 candidate.)
- **Native Compare command**: no API hook exists; use the documented Snapshot compare graph (same result, plus it works headless).
- **Direct BIMcollab / Newforma Konekt / Revizto / ACC Coordination sync**: these clouds only expose proprietary REST APIs. BCF 2.1 files (`BCF.ExportIssues`/`ImportIssues`) are the vendor-neutral bridge all of them accept — exactly as anticipated.
- **Deleting individual elements from the tree**: the API can only remove whole appended files (`Model.Remove`) — as expected. For anything finer, hide the items and publish an NWD.
- **Temporary transforms**: the 2024 API only offers permanent (undoable) transform overrides; there is nothing temporary to expose.
- **.xls (legacy BIFF) files, PDF reports, Appearance Profiler .dat files**: out of scope; use .xlsx and the HTML/CSV reports.
- **Deferred to the v0.4 backlog**: Export.SimulationImages (4D frame export), COBie.Export composite (primitives ship in v0.3), search-set XML import/export, glTF/OBJ mesh export, in-process multi-document (DocumentControl).
